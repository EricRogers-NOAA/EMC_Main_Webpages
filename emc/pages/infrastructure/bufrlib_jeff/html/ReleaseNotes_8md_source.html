59 ### Version 11.3.0 - May 21, 2019
60 
61 * A bug was fixed in the bitmap processing, so that each Table A mnemonic in
62 the jump/link table now tracks its own individual set of underlying Table C
63 operators. This involved changes to function igetrfel() and subroutines tabsub()
64 and makestab().
65 
66 * The build script makebufrlib.sh was modified to remove calls to cpp (the C
67 preprocessor) for all *.F Fortran files. Preprocessing for these files is now
68 done by fpp (the Fortran preprocessor), which is automatically called by the
69 Fortran compiler for all *.F files.
70 
71 * Global variable MXRST (the maximum number of long character strings that
72 can be read from a subset of a compressed BUFR message) has now been added to
73 the list of array size limits that can be configured using dynamic allocation.
74 
75 * The build script makebufrlib.sh was modified to stop generating the s_64
76 ("supersize") build. Such builds are now obsolete, since users can use the
77 dynamic allocation builds to flexibly define whatever size limits they need
78 for large BUFR datasets. If users need to recreate the exact specifications
79 that were present in the previous s_64 build, they can do so using a dynamic
80 allocation build and separate calls to subroutine isetprm() to set MXLCC=12,
81 MAXMEM=75000000, MXCDV=50000, MAXJL=128000, MAXSS=150000, MXMSGL=2500000 and
82 MXRST=500 within their application program.
83 
84 * A new capability was added to read and process master code and flag tables.
85 This feature is activated via an initial call to new subroutine codflg() at any
86 time after the first call to subroutine openbf(). Once that is done, then for
87 any BUFR message read into the library via subroutine readmg(), readerme() or
88 equivalent, and for any mnemonic contained within that message which is defined
89 as a code or flag table, the user may call new subroutine getcfmng() with that
90 mnemonic and an associated value (code figure or bit number) as input, and
91 the subroutine will return the meaning (as a character string) corresponding
92 to that mnemonic and associated value.
93 
94 * The default BUFR master table version number was changed from "13" to "29"
95 within subroutines cmsgini(), dxmini() and msgini().
96 
97 * The maximum number of delayed replication factors that can be passed in as
98 input to subroutine drfini() was increased from 200 to 2000.
99 
100 * Subroutines msgupd() and cpyupd() were modified so that the IPRT verbosity must
101 be greater than or equal to 1 in order for diagnostic alerts to be printed
102 whenever BUFR subsets greater than 65535 bytes are written to their own output
103 messages.
104 
105 * Function i4dy() was modified to change the window for converting 2-digit years
106 to 4-digit years. The old window was 1921-2020, i.e. add 1900 to any 2-digit
107 year greater than 20; otherwise add 2000. The new window will be 1941-2040,
108 i.e. add 1900 to any 2-digit year greater than 40; otherwise add 2000.

310 ### Version 10.2.0 - October 19, 2012
311 
312 * The makebufrlib.sh script was modified to streamline the endianness check
313 and make it more portable.
314 
315 * Subroutine wrtree() was modified to ensure that "missing" character strings
316 are properly encoded with all bits set to 1.
317 
318 * A new function icbfms() was added which tests whether decoded character
319 strings are "missing" by checking if all of the equivalent bits are set to 1.
320 This was done because, on certain platforms, the BUFRLIB REAL*8 "missing" value
321 BMISS is not always equivalent to all bits set to 1 when viewed as a character
322 string, and thus the existing BUFRLIB function ibfms() did not always work
323 properly in such cases. However, users can continue to use the existing IBFMS
324 function in application programs, because the new icbfms() function has now been
325 incorporated internally within the logic of many BUFRLIB subroutines, in
326 addition to also being available for direct calling by application programs.
327 
328 * Subroutines readmg() and readerme() were modified to prevent the BUFRLIB from
329 internally adjusting to DX (dictionary) table messages when Section 3 decoding
330 is being used. Otherwise, contention can occur between the table information
331 in the DX messages and the table information specified within the Section 3
332 descriptors. From now on, whenever Section 3 decoding is used (as specified
333 by setting IO="SEC3" when opening a file via openbf()), the BUFRLIB will now
334 treat any DX (dictionary) table message the same as any other message and
335 decode the actual data (i.e. table) values according to Section 3.
336 
337 * Subroutine openbf() was modified to allow a new option for input call
338 argument IO. If this argument is set to 'INUL', then the BUFRLIB will behave
339 the same as when IO='IN', except that it will never try to actually read
340 anything from the file attached to input call argument LUNIT. This can be
341 useful for some special cases, such as when the user plans to pass input
342 messages to the BUFRLIB using subsequent calls to subroutine readerme().
343 
344 * A new subroutine gettagpr() was added which returns the mnemonic corresponding
345 to a parent sequence in a subset definition, given the mnemonic corresponding
346 to a child descriptor within that sequence. This can be useful in certain
347 application codes, especially when Section 3 decoding is being used.
348 
349 * A new function getvalnb() was added which searches for a specified mnemonic
350 in a subset definition, then searches forward or backward from that point for
351 a different mnemonic and returns the associated value. This can be useful in
352 certain application codes, especially when Section 3 decoding is being used.
353 
354 * Functionality was added to improve the portability of reading and writing
355 BUFR messages across different platforms. All calls to existing FORTRAN
356 subroutines which read or write BUFR messages from disk (e.g. readmg(), ufbmem(),
357 ufbtab(), writsb(), wrcmps(), copymg(), etc.) now use embedded C-language I/O to
358 perform these tasks. Among other things, this means that any BUFR file can
359 now be read regardless of whether it has been pre-blocked with FORTRAN
360 control words using the cwordsh utility. For writing BUFR files, a new
361 subroutine setblock() was added which allows users to specify whether output
362 BUFR messages are to be unblocked (which is the new default), big-endian
363 blocked, or little-endian blocked.
364 
365 * A new subroutine setbmiss() was added which allows users to specify a custom
366 "missing" value for writing to and reading from BUFR files, rather than using
367 the BUFRLIB default "missing" value of 10E10. A corresponding function
368 getbmiss() was also added which returns the current "missing" value in use.

508 * Changes were made so that the BUFRLIB will automatically read and adjust
509 to any DX table (dictionary) messages internal to a file. Previously, the
510 software would only ever process such messages at the beginning of a file,

512 to these initial dictionary messages, and any subsequent dictionary messages

514 will cause the BUFRLIB to adjust its internal processing tables and treat all
515 subsequent data messages as conforming to these new dictionary messages, up
516 through the end of the file or until yet another set of dictionary messages
517 is encountered. These changes affect all BUFRLIB subroutines which read BUFR
518 messages from a file, including readmg(), ireadmg(), readmm(), ireadmm(), rdmemm(),

1559 <u>Y2K Compliance</u>
1560 
1561 Y2K compliance in the BUFR Archive Library is downwardly compatible. That is,
1562 the new library will read non-Y2K BUFR files as the old one does. However, all
1563 two digit years read are represented internally as four digit years, and any
1564 files written with the new library will be in Y2K format. A functional
1565 conversion of two digit year inputs assumes the years 21-99 are in the
1566 twentieth century, while years numbered 00-20 are in the twenty-first. A Y2K
1567 BUFR file is identified by a non-zero value in the 18th byte of the message
1568 section one, the century byte. At this point users of the library have access
1569 to the full four digit year values read by including a signal in their programs
1570 via a new entry point called DATELEN. The plan is to have the default set to
1571 return two digits of the year during a transition period. This allows
1572 implementation of the new BUFR Archive Library into a non-Y2K compliant
1573 environment. The susbsequent list of subroutines have been changed for Y2K
1574 compliance: BFRINI, DATEBF, DUMPBF, MSGINI, OPENMB, OPENMG, RDMEMM, READERME,
1575 READFT, READMG, READTJ.
