457 As noted during the discussion of subroutine openbf(), every BUFR file that is presented to the
458 BUFRLIB software, either for input (reading/decoding) or output (writing/encoding) purposes, must
459 have DX BUFR tables associated with it, unless the 'SEC3' decoding option is specified during the
460 call to openbf(). For all other cases, DX table information must be pre-defined and made available
461 to the software via call argument LUNDX during the call to openbf(). In the case of an existing
462 BUFR file, the DX tables information may be embedded within the first few BUFR messages of the
463 file itself. Otherwise, a separate ASCII text file containing the necessary DX tables information
464 must be supplied, such as the example shown above. It is extremely important that any such file
465 not only be syntactically correct but also complete, in the sense that all necessary mnemonics
466 must exist and be fully-defined.
467 
468 <br>
469 
470 First, let's define what we mean by a <i>mnemonic</i>. In short, a mnemonic is simply a
471 descriptive, alphanumeric name for an data value. In the context of the BUFRLIB software, there
472 are "Table A mnemonics", which refer to particular data subset (i.e. report ) types, "Table B
473 mnemonics", which refer directly to basic data values, and "Table D mnemonics", which are
474 sequences composed of one or more Table B (or other Table D!) mnemonics and which are themselves
475 normally direct constituents of a particular Table A mnemonic. In other words, at the highest level,
476 we have a Table A mnemonic which completely describes a type of data subset (e.g. rawinsonde,
477 wind profiler, etc.), and this Table A mnemonic is defined as a sequence of one or more Table B or
478 Table D mnemonics, where each Table D mnemonic is likewise itself defined as a sequence of one or
479 more Table B or Table D mnemonics, and so on until the entire data subset can be equivalently
480 described as a sequence of one or more Table B mnemonics which, again, themselves correspond to
481 basic data values (e.g. pressure, temperature, humidity, etc.). In this way, the entire sequence
482 of data values that constitute a particular type of data subset is fully and unambiguously defined,
483 both for purposes of input (reading/decoding) or output (writing/encoding) of reports corresponding
484 to that particular type of data subset.
485 
486 <br>
487 
488 However, it is also important to understand what mnemonics are not. Specifically, mnemonics never
489 themselves appear within actual BUFR messages that are read or written by the BUFRLIB software;
490 rather, their only purpose in life is to make it easier for users to interact with the software by
491 providing descriptive names to represent individual data values, as opposed to having to keep track
492 of the corresponding FXY numbers (described below), which are much less intuitive but which
493 nevertheless are the prescribed method within the BUFR code form for referencing of individual data
494 values, and which therefore are what are actually read and written by the software.
495 
496 <br>
497 
498 As we begin our actual discussion of BUFR tables files, let's start with an overview, by noting
499 that a BUFR tables file consists of three distinct sections, each of which contains one or more
500 lines of 80 characters in length, and where a "*" as the first character of a line indicates that
501 that entire line is a comment. In the first section, all Table A, B and D mnemonics that are to be
502 used within the file are initially declared, assigned a unique FXY number, and given a short,
503 free-form text description. Then, in the second section, all previously-declared Table A and
504 Table D mnemonics are actually defined as a sequence of one or more Table B (or other Table D!)
505 mnemonics. Finally, in the third section, all previously-declared Table B mnemonics are defined
506 in terms of their scale factor, reference value, bit width, and units.
507 
508 <br>
509 
510 Now, as we delve into the details of each of the three sections, we'll constantly refer back to
511 our [sample DX BUFR tables file](#bftab) in order to better illustrate the concepts that are discussed.
512 
513 ### Section 1
514 
515 As previously mentioned, the first section of a BUFR tables file is where all Table A, B and D
516 mnemonics are initially declared, assigned a unique FXY number, and given a short free-form text
517 description. Mnemonics may contain any combination of uppercase letters and numbers (or, in certain
518 special cases, a "." character!), up to a maximum total of 8 characters in length. A mnemonic may be
519 declared only once, and each one must correspond to a unique FXY number, which itself consists of 6
520 characters, and where the first character (i.e. the "F" component) is an "A" if the mnemonic is
521 being declared as a Table A mnemonic, "3" if the mnemonic is being declared as a Table D mnemonic,
522 and "0" if the mnemonic is being declared as a Table B mnemonic. Otherwise, the remainder of the FXY
523 number must be all digits, with the next 2 characters (i.e. the "X" component) as a number
524 between 00 and 63, and the final 3 characters (i.e. the "Y" component) as a number between 001
525 and 255. Readers who are more familiar with BUFR will immediately recognize these F, X, and Y
526 values as those that are defined within the
527 [official documentation of the BUFR code form](https://library.wmo.int/index.php?lvl=notice_display&id=10684#.X68yu8hKiUn);
528 therefore, by international convention, a mnemonic should not be given an X value between 00 and 47
529 along with a Y value between 001 and 191 unless that mnemonic, when subsequently defined, corresponds
530 exactly to the BUFR descriptor having that same FXY number within the official,
531 internationally-coordinated WMO BUFR tables. For example, in our [sample DX BUFR tables file](#bftab),
532 mnemonic "WMOB" is declared with an FXY number of 001001; therefore, it has the exact same text
533 description (i.e. "WMO BLOCK NUMBER") and, when later defined within the last section of the file,
534 the exact same scale factor, reference value, bit width, and units as for FXY number 001001 within
535 the official WMO BUFR tables. This concept should be somewhat intuitive, but it's obviously very
536 important when the BUFRLIB software is to be used to encode BUFR messages that may potentially be
537 read by other users in other organizations around the world.
538 
539 <br>
540 
541 In looking further at our [sample DX BUFR tables file](#bftab), we see that the lines within the first
542 section each contain a "|" character in columns 1, 12, 21, and 80. Mnemonics are declared, and are
543 left-justified, in columns 3-10, corresponding FXY numbers are assigned in columns 14-19, and the
544 corresponding text description begins in column 23. All of the Table A mnemonics are declared first,
545 followed by all of the Table D mnemonics, followed by all of the Table B mnemonics. Within each set,
546 it is generally a good idea for human-readability purposes to list the mnemonics in ascending order
547 with respect to their FXY number, although this is not required by the BUFRLIB software itself.
548 Likewise, human-readability can usually also be improved by the judicious use of one or more
549 "separator" lines containing the required "|" character in columns 1, 12, 21, and 80 but without
550 any actual mnemonic declaration; however, again, the use of such "separator" lines is not required
551 by the software. In fact, the software will simply continue reading lines of the file, one at a time,
552 and looking for new mnemonic declarations, until it reaches a line which does not contain a "|"
553 character in each of columns 1, 12, 21, and 80, at which point it then knows that the first section
554 of the tables file has ended.
555 
556 <br>
557 
558 We mentioned earlier that mnemonics only exist in order to facilitate user interaction with the
559 BUFRLIB software and that, therefore, mnemonics should be as intuitive as possible. We now need
560 to amend that statement slightly, because certain Table A mnemonics do have a special additional
561 function. Specifically, if a Table A mnemonic consists of 8 characters (i.e. the maximum) and if
562 characters 3 through 8 are all digits, then the mnemonic is also used by the software to set the
563 data category and local subcategory within Section 1 of each BUFR message when writing/encoding
564 data subsets corresponding to that mnemonic. In such cases, characters 3 through 5 define the
565 category, and characters 6 through 8 define the subcategory. Therefore, in referring again to our
566 [sample DX BUFR tables file](#bftab) where we've defined three different Table A mnemonics,
567 we've also indicated that, e.g. when we use the software to write/encode data subsets according
568 to the Table A mnemonic "NC002007" (i.e. wind profiler), we want all BUFR messages which contain
569 such data subsets to be encoded as category 2 and local subcategory 7 within Section 1 of the
570 message.
571 
572 <br>
573 
574 Incidentally, even if a Table A mnemonic doesn't meet the above criteria, BUFR message category
575 and local subcategory values will still be set by the software when writing/encoding BUFR data
576 subsets corresponding to that Table A mnemonic. However, in such cases, the category value will
577 be set to the "Y" component (i.e. last 3 digits) of the FXY number corresponding to the mnemonic,
578 and the subcategory value will simply be set to 0. Therefore, it is recommended to use the
579 previous, more-explicit approach when assigning a Table A mnemonic for a data subset to be output,
580 since this approach provides for greater control over the category and subcategory values that
581 will be encoded into Section 1 of the resultant BUFR message. We should also take this opportunity
582 to point out that, when the FXY number corresponding to a Table A mnemonic is actually encoded
583 into a BUFR message, a "3" is actually encoded in place of the "A" which is used in the DX tables
584 file. Put another way, the "A" that appears within the FXY number corresponding to each Table A
585 mnemonic within the tables file is only there so that such mnemonics can be easily distinguished
586 from Table D mnemonics by the software.
